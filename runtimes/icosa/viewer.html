🔷 Icosa Gallery + AR viewer

stub — NOT a runtime; CONSUMER integration

Scope

Icosa is not an XRAI rendering engine — it's a gallery + API for 3D assets. XRAI's object.glb / object.tilt entities can reference Icosa-hosted assets; Icosa's own web AR viewer renders the glTF/tilt payload; XRAI adds the relational + temporal metadata on top.

Integration paths

  • Publish: export an XRAI scene's object.glb to Icosa Gallery via the vendored com.icosa.icosa-api-client-unity package (in the Portals monorepo). See spec 009 Paint-AR migration plan § "the 4-layer Icosa divide".
  • Embed: Icosa's public AR viewer URL can be embedded in an iframe below an XRAI doc — the asset renders in Icosa's viewer, XRAI-specific metadata displays alongside.
  • Import: Icosa Gallery GLB URL → reference in object.glb.model_url → any XRAI runtime loads it.
  • Paint brushes: AllBrushes.asset (72 Open Brush brushes) ships with the vendored package — curated 10 for our Paint-AR layer per spec 009 migration plan v2.

Status

Consumer-side integration documented. No dedicated viewer page needed — Icosa is the viewer for its own content. Our role: publish XRAI entities that reference Icosa assets, never embed Icosa inside our render loop.

Why document this?

Launch-day FAQ (FAQ.md) had to clarify "is Icosa a runtime?" since xra1 and imclab/xrai both trade on the XRAI name. Short answer in RUNTIMES.md § What Icosa IS and IS NOT. This page is the engine-picker target that redirects to that clarification.
